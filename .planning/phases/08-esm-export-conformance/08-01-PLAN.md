# Phase 8: ESM Export Conformance

**Problem:** Our modules were designed for CJS-style `_registerBuiltinOverride('fs', fsObject)`. But Claude Code (and Codex/Gemini) use ESM imports: `import { readFileSync } from "node:fs"`. The browser resolves these via import maps to our module files, which must export the exact named exports that Node.js provides.

**Fix:** Every module file must export named exports matching the real Node.js module shape. This is not optional — a single missing export crashes the CLI at load time.

**Approach:** For each module, I extracted every named import from Claude Code's `cli.js` (12MB bundled). This is the minimum required set. For completeness, I also list additional exports the module should have for Codex and Gemini compatibility.

---

### 1. `napi-bridge/fs.js`

**Claude Code imports:**
```js

         openSync, readFileSync, realpathSync, unlinkSync } from "node:fs"
```

```js
export function readdirSync(path, options) { ... }
export function readdir(path, options, cb) { ... }
export function statSync(path) { ... }
export function stat(path, cb) { ... }
export function lstatSync(path) { ... }
export function lstat(path, cb) { ... }
export function mkdirSync(path, options) { ... }
export function mkdir(path, options, cb) { ... }
export function unlinkSync(path) { ... }
export function unlink(path, cb) { ... }
export function renameSync(oldPath, newPath) { ... }
export function rename(oldPath, newPath, cb) { ... }
export function existsSync(path) { ... }
export function accessSync(path, mode) { ... }
export function access(path, mode, cb) { ... }
export function realpathSync(path) { ... }
export function realpath(path, cb) { ... }
export function createReadStream(path, options) { ... }
export function createWriteStream(path, options) { ... }
export function openSync(path, flags, mode) { ... }
export function open(path, flags, mode, cb) { ... }
export function closeSync(fd) { ... }

export function writeSync(fd, buffer, offset, length, position) { ... }

export function rmSync(path, options) { ... }
```

**Pattern:** Define each function as a standalone named export. Build the default export object from those same functions. This ensures both `import { readFileSync } from "node:fs"` and `import fs from "node:fs"` work.

## Dual-Path Compatibility

The modules must work in BOTH paths:

1. **Browser ESM path** (import map): `import { readFileSync } from "node:fs"` → resolves to `napi-bridge/fs.js` → uses named exports
2. **MEMFS require() path** (CJS bridge): `_registerBuiltinOverride('fs', fsModule)` → the registered object must have the same methods

**Solution:** Each module's default export IS the object that gets registered via `_registerBuiltinOverride`. Named exports are destructured from the same object. Both paths get the same implementation.

```js
// Pattern for every module:
export function readFileSync(...) { ... }
export function writeFileSync(...) { ... }
// ... all named exports

const fs = { readFileSync, writeFileSync, ... };
export default fs;

// In browser-builtins.js:
import fs from './fs.js';
_registerBuiltinOverride('fs', fs);  // CJS path gets the same object
```
