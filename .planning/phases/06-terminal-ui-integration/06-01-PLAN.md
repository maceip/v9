## Implementation Steps

### Step 1: xterm.js Terminal Setup (TERM-01)

**What:** Embed xterm.js in a web page with proper rendering.
**Files:** New: `web/index.html`, `web/terminal.js`

```html
<div id="terminal"></div>
<script type="module">
  import { Terminal } from 'xterm';
  import { FitAddon } from 'xterm-addon-fit';
  import { WebLinksAddon } from 'xterm-addon-web-links';

  const term = new Terminal({
    cursorBlink: true,
    fontSize: 14,
    fontFamily: 'Menlo, Monaco, monospace',
    theme: { background: '#1a1b26' },
  });

  const fitAddon = new FitAddon();
  term.loadAddon(fitAddon);
  term.loadAddon(new WebLinksAddon());
  term.open(document.getElementById('terminal'));
  fitAddon.fit();
</script>
```

**ANSI support (TERM-01):**
xterm.js handles ANSI natively — colors, bold, italic, cursor movement, clear screen all work out of the box. No custom parsing needed.

### Step 2: Keyboard Input → process.stdin (TERM-02)

**What:** Wire xterm.js `onData` to `process.stdin` Readable stream.
**Files:** `web/terminal.js`, modify `napi-bridge/browser-builtins.js`

```js
// xterm → process.stdin
term.onData((data) => {
  runtime.pushStdin(data);
});

// In browser-builtins.js, processBridge.stdin needs a pushData method:
// stdin.push(Buffer.from(data)) feeds data to the Readable
```

The `process.stdin` Readable from Phase 3 needs a public `pushData()` method that the terminal can call. This was flagged in Phase 3 design — the stream exists but has no external input source until now.

### Step 3: process.stdout → xterm (TERM-01)

**What:** Wire `process.stdout` Writable to xterm.js `write()`.
**Files:** `web/terminal.js`

```js
// In initEdgeJS options:
const runtime = await initEdgeJS({
  onStdout: (text) => term.write(text),
  onStderr: (text) => term.write(text), // stderr also goes to terminal
});
```

This should already work via the `onStdout`/`onStderr` callbacks in `initEdgeJS`. Verify ANSI escape codes pass through unmodified.

### Step 4: Terminal Resize (TERM-03)

**What:** Handle browser window resize → update terminal dimensions → signal process.
**Files:** `web/terminal.js`

```js
window.addEventListener('resize', () => {
  fitAddon.fit();
});

term.onResize(({ cols, rows }) => {
  runtime.setTerminalSize(cols, rows);
  // Equivalent to SIGWINCH — process.stdout.columns/rows update
});
```

In `processBridge`:
```js
process.stdout.columns = cols;
process.stdout.rows = rows;
process.emit('SIGWINCH');
```

### Step 5: Module Resolution (INTG-01)

**What:** `require()` resolves built-in modules and relative paths.
**Files:** `napi-bridge/index.js` (extend module registration from Phase 3)

Module resolution order:
1. Check `_registerBuiltinOverride` map (events, fs, path, stream, etc.)
2. Check browser builtins (crypto, buffer, process, url, util)
3. For relative paths (`./foo`, `../bar`): resolve against MEMFS, load and eval
4. For package names (`@anthropic-ai/sdk`): look in `/node_modules/`

**Pre-bundling strategy:**
Claude Code + Anthropic SDK + dependencies are too large to load at runtime via `require()`. They should be pre-bundled (webpack/esbuild) into a single JS file that gets loaded into MEMFS.

```js
initEdgeJS({
  files: {
    '/app/bundle.js': bundledClaudeCodeSource,
    '/app/node_modules/...': sdkFiles,
  },
});
```

### Step 6: Anthropic SDK Loading (INTG-02)

**What:** The Anthropic SDK dependency tree loads and initializes.
**Key dependencies to verify:**
- `@anthropic-ai/sdk` — the SDK itself
- `node-fetch` or native `fetch` — HTTP client
- `form-data` — multipart uploads (may not be needed)
- `abort-controller` — request cancellation (browser-native)

**Strategy:** Bundle the SDK with all dependencies pre-resolved. Use esbuild with `--platform=browser --bundle` to eliminate Node.js-specific imports.

```bash
esbuild --bundle node_modules/@anthropic-ai/sdk/index.js \
  --platform=browser \
  --format=esm \
  --external:crypto \
  --outfile=dist/anthropic-sdk-bundle.js
```

Register the bundle as a built-in module override:
```js
_registerBuiltinOverride('@anthropic-ai/sdk', bundledSDK);
```

### Step 7: Full Conversation Flow (INTG-03)

**What:** User types prompt → Claude Code processes → API call → streaming response → tool use → result displayed.

The conversation flow:
```
1. User types in xterm.js
2. Keyboard data → process.stdin
3. Claude Code reads stdin, constructs API request
4. https.request() → fetch() proxy → Anthropic API
5. Streaming SSE response → IncomingMessage Readable
6. Claude Code processes response chunks
7. If tool_use: execute tool (file read/write via fs, bash via child_process)
8. Tool result → next API call
9. Final response → process.stdout → xterm.js
10. Loop back to 1
```

**Test this with a minimal conversation:**
```js
// Pre-populate with a simple project
const runtime = await initEdgeJS({
  env: {
    ANTHROPIC_API_KEY: 'sk-ant-...',
  },
  files: {
    '/project/hello.js': 'console.log("hello world");',
  },
});

// Simulate: "Read the file hello.js and tell me what it does"
// Claude Code should:
//   1. Call the file read tool
//   2. Read /project/hello.js from MEMFS
//   3. Return the explanation via API
//   4. Display in terminal
```

## Risks

| Risk | Severity | Mitigation |
|------|----------|-----------|
| SDK bundle size | MEDIUM | Tree-shake with esbuild; SDK is ~200KB minified |
| Module resolution complexity | HIGH | Pre-bundle everything; don't try to implement full `require()` resolution |
| CORS on Anthropic API | HIGH | May need CORS proxy (Cloudflare Worker). Assess in Phase 4. |
| COOP/COEP headers for SharedArrayBuffer | MEDIUM | Only needed if Phase 5 uses Atomics; not needed for shim-only approach |
| xterm.js size | LOW | ~265KB; acceptable for this use case |
| First-load latency | MEDIUM | 7MB Wasm + SDK bundle + xterm.js = ~10MB total; use streaming instantiation |
