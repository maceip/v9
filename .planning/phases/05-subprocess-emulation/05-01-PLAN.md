### Step 1: child_process Module Override

**What:** Replace Node.js `child_process` with browser implementation.
**File:** New: `napi-bridge/child-process.js`

```

```

**ChildProcess** (extends EventEmitter):
```

child.stdout     â†’ Readable stream

child.pid        â†’ number (fake, monotonic counter)
```

### Step 2: Shell Command Parser

**What:** Parse `bash -c 'command'` and extract the actual command.
**File:** `napi-bridge/shell-parser.js`

Handle:
- `bash -c 'ls -la /path'` â†’ `{ cmd: 'ls', args: ['-la', '/path'] }`
- `sh -c 'command'` â†’ same
- Direct: `ls /path` â†’ `{ cmd: 'ls', args: ['/path'] }`
- Pipes: `cat file | grep pattern` â†’ chain of commands
- Redirects: `echo hello > file` â†’ capture output to file
- Environment: `FOO=bar command` â†’ set env then run
- Semicolons: `cd /path; ls` â†’ sequential execution
- `&&` / `||`: conditional chaining
- Quoting: `'single'`, `"double"`, `\escape`
- Globbing: `*.js` â†’ expand against MEMFS (basic)

**NOT supported (throw informative error):**
- Backgrounding (`&`)
- Subshells (`$(...)`, backticks)
- Here-docs (`<<EOF`)
- Complex bash features (arrays, functions, loops)

### Step 4: Git Shims (PROC-05)

**What:** Read-only git operations against a virtual git state.
**File:** `napi-bridge/git-shim.js`

Since MEMFS has no real git repo, git shims operate on a virtual git state that can be pre-populated via `initEdgeJS({ git: { ... } })`.

| Command | Implementation |
|---------|----------------|
| `git status` | Compare MEMFS state against initial file snapshot |
| `git log --oneline -N` | Return pre-populated commit history |
| `git diff` | Diff current MEMFS vs initial snapshot |
| `git diff --name-only` | List changed files |
| `git blame FILE` | Return pre-populated blame data or stub |
| `git branch` | Return current branch name |
| `git rev-parse HEAD` | Return HEAD commit hash |

**Pre-population:**
```js
initEdgeJS({
  git: {
    branch: 'main',
    head: 'abc123',
    log: [
      { hash: 'abc123', message: 'Initial commit', date: '2026-03-19' },
    ],
    initialFiles: { '/project/index.js': 'original content' },
  }
});
```

### Step 7: Web Worker Fallback (PROC-01, PROC-02, PROC-03)

**What:** For commands that don't have shims, spawn a Web Worker.
**File:** `napi-bridge/worker-process.js`

This is the slow path â€” most commands should hit shims first.

```
spawn('custom-tool', ['arg1']) â†’
  1. Create Web Worker
  2. Load Wasm instance in worker
  3. Wire stdio streams between main thread and worker
  4. Run command in worker
  5. Capture exit code via message
  6. Emit 'exit' and 'close' events
```

**Note:** This requires SharedArrayBuffer (COOP/COEP headers) or postMessage-based stdio. postMessage is simpler but adds latency.

## Risks

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Shell parser complexity | HIGH | Don't build a full bash parser; handle the patterns Claude Code actually uses |
| execSync blocking | MEDIUM | Shims are synchronous by nature; Worker path needs Atomics.wait or sync XMLHttpRequest |
| Git shim fidelity | LOW | Claude Code is tolerant of git failures; stubs that return "not a git repo" are acceptable |
| Glob expansion against MEMFS | MEDIUM | Simple `*.ext` matching only; no `**` or `{a,b}` |
